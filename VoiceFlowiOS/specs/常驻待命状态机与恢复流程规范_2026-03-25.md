# VoiceFlow 常驻待命状态机与恢复流程规范

更新时间：2026-03-25  
适用范围：`VoiceFlowiOS` 主 App、`VoiceFlowKeyboard` 键盘扩展、跨进程状态同步、Live Activity / 灵动岛展示、录音恢复链路  
目标读者：Codex / Claude Code / iOS 工程师

---

## 1. 目标

本规范定义 VoiceFlow 从“单次录音工具”切换为“常驻待命语音输入服务”后的统一状态机、交互规则、跨进程同步规则与日志要求。

本规范要解决的问题：

- 停止录音后不应默认关闭 VoiceFlow，而应回到待命态
- 用户手动关闭 VoiceFlow 后，从键盘点录音应能自动恢复
- 系统麦克风权限不可用时，应跳主 App 做权限恢复，而不是静默失败
- `processing` 期间应统一阻止再次录音
- Live Activity / 灵动岛应有稳定、明确、可预测的展示规则
- 主 App 与键盘扩展必须共享同一套服务状态语义，避免“主 App 认为开启、键盘认为关闭”这类分裂状态

非目标：

- 本规范不定义 UI 视觉细节的像素级样式
- 本规范不要求实现实时流式字幕
- 本规范不要求本轮解决所有 ASR / LLM 延迟优化问题

---

## 2. 核心原则

1. “停止录音”只结束本次 utterance，不等于关闭 VoiceFlow。
2. VoiceFlow 默认运行在“常驻待命”模型下，除非用户主动关闭或命中自动关闭策略。
3. 业务开关关闭与系统麦克风权限关闭是两类完全不同的问题，必须分开处理。
4. 键盘侧点击录音时，所有恢复逻辑都必须可预测，不能静默失败。
5. 结果回填必须基于“本次 session 的来源”判断，不能只靠一个全局布尔值临时猜测。
6. Live Activity 的存在是预期行为，不再把“停止录音后立刻消失”作为默认目标。

---

## 3. 统一术语

- `VoiceFlow`：业务层语音输入服务，不等于系统麦克风权限
- `service state`：VoiceFlow 服务状态
- `armed`：常驻待命态，可快速开始下一次录音
- `recording`：当前正在录音
- `processing`：录音已停止，正在做 ASR / LLM / 回填
- `disabledByUser`：用户主动关闭了 VoiceFlow 业务开关
- `disabledBySystemPermission`：系统麦克风权限不可用
- `session source`：本次录音来源，至少区分 `keyboard` / `mainApp`

---

## 4. 服务状态机

### 4.1 定义

| 状态 | 含义 | 是否显示 VoiceFlow 待命型灵动岛 | 是否允许快速开始下一次录音 |
|---|---|---:|---:|
| `disabledByUser` | 用户主动关闭了 VoiceFlow 业务开关 | 否 | 否 |
| `disabledBySystemPermission` | 系统麦克风权限不可用 | 否 | 否 |
| `armed` | VoiceFlow 已开启，常驻待命，可快速进入录音 | 是 | 是 |
| `recording` | 正在录音 | 是 | 不适用 |
| `processing` | 已停止录音，正在做 ASR / LLM / 回填 | 是 | 否 |

### 4.2 状态切换

| 触发事件 | 当前状态 | 目标状态 | 备注 |
|---|---|---|---|
| 用户在主 App 打开 VoiceFlow | `disabledByUser` | `armed` | 业务开关恢复 |
| 用户在主 App 关闭 VoiceFlow | `armed` / `processing` / `recording` | `disabledByUser` | 必须释放资源 |
| 系统麦克风权限被撤销 | 任意 | `disabledBySystemPermission` | 立即失效 |
| 系统权限恢复并业务开关开启 | `disabledBySystemPermission` | `armed` | 仅权限恢复完成后进入 |
| 开始录音 | `armed` | `recording` | 主 App 或键盘均可触发 |
| 用户停止录音 | `recording` | `processing` | 本次 utterance 结束 |
| ASR / LLM / 回填完成 | `processing` | `armed` | 默认回待命态 |
| 自动关闭计时到期 | `armed` | `disabledByUser` | 视为自动关闭服务 |

### 4.3 明确约束

- 禁止 `recording -> disabledByUser` 的隐式跳转，除非用户手动关闭。
- 禁止 `processing -> recording` 的并发切换。
- 禁止 `processing` 期间排队下一次录音。
- `processing` 完成后必须默认回到 `armed`，不能默认回 `disabledByUser`。

## 5. 键盘点击录音的决策树

键盘扩展点击录音按钮时，必须按以下优先级判断：

### 5.1 判断顺序

1. 系统麦克风权限是否可用
2. VoiceFlow 业务开关是否开启
3. 当前 `service state`

### 5.2 决策表

| 条件 | 行为 |
|---|---|
| `recordPermission != granted` | 自动跳主 App，进入“系统权限恢复”流程 |
| `recordPermission == granted && serviceState == disabledByUser` | 自动跳主 App，自动恢复 VoiceFlow 到 `armed`，提示用户返回键盘 |
| `serviceState == armed` | 直接开始录音 |
| `serviceState == recording` | 接管当前会话，键盘显示录音中 |
| `serviceState == processing` | 统一阻止，提示“上一段仍在处理中，请稍后” |
| 其他未知状态 | 视为异常，跳主 App 做恢复 |

### 5.3 禁止行为

- 禁止键盘在 `processing` 期间再次发起新的录音请求
- 禁止键盘因一个泛化错误文案就静默回 `idle`
- 禁止键盘在发起 start 前无条件清空服务状态源

## 6. 主 App 恢复流程

### 6.1 disabledByUser 恢复流程

适用条件：

- 系统麦克风权限仍然是 `granted`
- VoiceFlow 业务开关是关闭状态

预期行为：

1. 键盘点击录音
2. 自动跳主 App
3. 主 App 自动把 VoiceFlow 从 `disabledByUser` 恢复为 `armed`
4. 页面提示用户：“VoiceFlow 已恢复，请手动切换回语音输入键盘继续录音”
5. 不要求用户再手动点业务开关

### 6.2 disabledBySystemPermission 恢复流程

适用条件：

- 系统麦克风权限不是 `granted`

预期行为：

1. 键盘点击录音
2. 自动跳主 App
3. 主 App 进入“系统权限恢复”页
4. 如果权限状态是 `undetermined`，可触发系统授权弹窗
5. 如果权限状态是 `denied`，引导用户去系统设置
6. 权限恢复后，VoiceFlow 状态进入 `armed`
7. 提示用户手动切换回语音输入键盘继续录音

### 6.3 恢复后的要求

- 主 App 只负责恢复和提示，不自动尝试切回输入场景
- 用户手动切换回语音输入键盘

---

## 7. 自动关闭策略

### 7.1 支持项

- `5分钟后关闭`
- `1小时后关闭`
- `永不关闭`

### 7.2 默认值

- 默认 `5分钟后关闭`

### 7.3 生效范围

- 只作用于 `armed`
- 不作用于 `recording`
- 不作用于 `processing`

### 7.4 计时规则

- `processing -> armed` 后开始计时
- 在 `armed` 期间若用户再次录音，计时重置
- 到期后转为 `disabledByUser`

---

## 8. 灵动岛 / Live Activity 规则

### 8.1 展示规则

在以下状态下，允许显示 VoiceFlow 自定义灵动岛内容：

- `armed`
- `recording`
- `processing`

在以下状态下，不显示待命型 VoiceFlow 灵动岛：

- `disabledByUser`
- `disabledBySystemPermission`

### 8.2 内容规则

Live Activity / 灵动岛自定义内容只显示：

- VoiceFlow 的 logo 图案主体

不显示：

- 外框
- 状态文案
- 按钮
- 其他附加文本

### 8.3 系统限制说明

- 上述规则仅针对应用自定义的 Live Activity / 灵动岛内容
- iOS 系统级麦克风隐私指示不可自定义替换

---

## 9. 录音与结果回填规则

### 9.1 录音结束后的行为

- 用户点击停止后，结束本次 utterance
- 服务状态进入 `processing`
- 转录、润色、结果回填完成后，回到 `armed`

### 9.2 结果回填

必须满足：

1. 结果回填依据使用 `session source`，而不是仅依赖一个全局 `isKeyboardRecording`
2. 只要本次 session 来源是键盘，就必须尝试把结果回填给键盘
3. 结果投递链必须具备以下阶段日志：
   - 结果生成
   - `pendingResult` 写入
   - 读回校验
   - `resultReady` 通知发送
   - 键盘收到通知
   - 键盘读取结果
   - `textDocumentProxy.insertText()` 调用

### 9.3 processing 期间的行为

- 若用户再次点录音，统一阻止
- 文案提示：
  - “上一段仍在处理中，请稍后”

---

## 10. 错误码与提示文案

错误码必须至少区分：

| 错误码 | 语义 | 键盘侧处理 |
|---|---|---|
| `needs_system_permission` | 系统麦克风权限不可用 | 自动跳主 App，进入权限恢复 |
| `needs_voiceflow_reenable` | VoiceFlow 被用户手动关闭 | 自动跳主 App，自动恢复并提示返回 |
| `busy_processing` | 上一段仍在处理中 | 阻止再次录音 |
| `asr_timeout` | ASR 超时 | 显示语音识别超时 |
| `llm_timeout` | LLM 超时 | 显示润色超时 |
| `inject_failed` | 键盘回填失败 | 显示回填失败 |
| `generic` | 未分类错误 | 通用错误提示 |

禁止继续使用一个笼统的：

- “处理超时，请重试”

来覆盖所有失败路径。

---

## 11. App Group / 跨进程状态同步

### 11.1 必须有统一的服务状态源

共享存储必须至少暴露：

- `serviceState = disabledByUser / disabledBySystemPermission / armed / recording / processing`
- `voiceFlowEnabled`
- `recordPermissionSnapshot`
- `pendingResult`
- `recordingState`（若保留，必须与 `serviceState` 语义对齐）
- `sessionSource`

### 11.2 禁止行为

- 禁止键盘在 start 前无条件删除 `recordingState`
- 禁止主 App 和键盘各自推断状态，不共享同一个服务状态源

---

## 12. 必加日志

以下日志为实现本规范时的最低要求。

### 12.1 start 决策日志

- `serviceState`
- `voiceFlowEnabled`
- `recordPermission`
- `applicationState`
- `decision=direct_start / jump_permission / jump_reenable / busy_processing`

### 12.2 stop / processing 日志

- 用户点击停止
- 录音停止完成
- 进入 `processing`
- 处理完成回到 `armed`
- 自动关闭计时开始 / 重置 / 到期

### 12.3 回填日志

- `sessionSource`
- `pendingResult write`
- `pendingResult read-back`
- `resultReady posted`
- `KeyboardVM onResultReady`
- `insertText called`
- `insertText completed`

### 12.4 timeout 日志

必须分开：

- `asr_timeout`
- `llm_timeout`
- `inject_timeout`
- `processing_timeout`

---

## 13. 实施约束

1. 先落状态机和决策树，再改具体录音实现细节。
2. 先保证“状态语义正确”，再优化“延迟和无感体验”。
3. 禁止在没有统一 service state 的情况下继续堆补丁。
4. 后续任何录音链路改动都必须遵守本规范的状态定义。

---

## 14. 当前决议摘要

本轮需求已确认如下：

- VoiceFlow 采用常驻待命模型
- 停止录音后回到 `armed`
- 用户手动关闭后进入 `disabledByUser`
- 键盘在 `disabledByUser` 下点录音时，自动跳主 App 并自动恢复
- 系统权限不可用时，跳主 App 做权限恢复
- 恢复后只提示用户手动切回语音输入键盘
- `processing` 期间统一阻止再次录音
- 自动关闭默认值为 `5分钟后关闭`
- 灵动岛自定义内容只显示 VoiceFlow logo 图案主体
